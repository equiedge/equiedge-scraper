TIER OVERVIEW
--------------------------------------------------------------------------------

  Trial (Free)
    - 3 free track-day analyses
    - Expires after 3 days from account creation OR 3 track-days used (whichever first)
    - No payment required, but Sign in with Apple required

  Basic — $14.99/month ($11.99/month billed annually at $143.88/yr)
    - 10 track-day analyses per week (resets Monday)
    - All races within an analysed track included
    - Re-analysing the same track on the same day does NOT count again

  Pro — $34.99/month ($28.99/month billed annually at $347.88/yr)
    - Unlimited track-day analyses
    - All tracks, all races

  Expired (no subscription, trial over)
    - Hard paywall on Edge AI analysis
    - Can still browse race listings, runner fields, TAB odds (app is not blank)


AUTHENTICATION — SIGN IN WITH APPLE
--------------------------------------------------------------------------------

  Why:
    - Prevents trial abuse (reinstall/new device = same Apple ID = same account)
    - Ties subscription status to a server-side user record
    - Minimal friction (one tap, no email/password)
    - Apple favours apps that use Sign in with Apple during review

  Flow:
    1. First launch -> Sign in with Apple prompt (required before any use)
    2. Apple returns a stable user identifier (per-app, per-Apple-ID)
    3. App sends identifier to Vercel backend -> creates or retrieves user record
    4. Backend returns: { userId, tier, trialStartDate, trialUsage, weeklyUsage }
    5. Subsequent launches: cached userId in Keychain, validated against backend

  User record (stored on Vercel backend):
    - appleUserId: String (stable identifier from Sign in with Apple)
    - tier: "trial" | "basic" | "pro" | "expired"
    - trialStartDate: ISO date (set on first sign-in)
    - trialUsage: ["randwick_2026-04-13", ...] (max 3)
    - weeklyUsage: { isoWeek: "2026-W16", tracks: ["randwick_2026-04-13", ...] }
    - subscriptionExpiresAt: ISO date (from App Store Server Notifications)
    - createdAt: ISO date

  Why Keychain (not UserDefaults):
    - Survives app deletion/reinstall on same device
    - Cannot be cleared by user without full device reset
    - Extra layer of trial abuse prevention


TRACK-DAY COUNTING RULES
--------------------------------------------------------------------------------

  What counts as "1 track-day":
    - Running analysis on "Randwick" on Monday = 1
    - Running analysis on "Randwick" on Tuesday = 2 (different day)
    - Re-running "Randwick" on Monday (refresh) = still 1 (same track-day)

  Storage:
    - Server-side on user record (authoritative)
    - Composite key format: "randwick_2026-04-13"
    - Basic: weeklyUsage.tracks.count < 10 OR key already exists (free re-run)
    - Trial: trialUsage.count < 3 AND now <= trialStartDate + 3 days
    - Week resets every Monday (based on ISO week number)


SERVER-SIDE CACHE — CONSISTENT AI PICKS
--------------------------------------------------------------------------------

  Problem:
    - Multiple users requesting the same track-day triggers redundant scrapes
    - Grok is non-deterministic: different users could get different horse picks
    - Concurrent scrapes waste Grok API spend and risk FormFav rate-limiting

  Solution: Cache first analysis per track-day on the backend
    - Cache key: "randwick_2026-04-13"
    - Cache value: full JSON (scraped race data + Grok AI analysis)
    - TTL: end of day (or ~18 hours)
    - Storage: Vercel KV (Redis) or Upstash Redis (free tier generous)

  Flow:
    1. User requests analysis for Randwick on 2026-04-13
    2. Backend checks cache for key "randwick_2026-04-13"
       HIT  -> Return cached result instantly (no scrape, no Grok call)
       MISS -> Scrape FormFav, call Grok, store in cache, return result
    3. All subsequent users that day get identical picks

  Benefits:
    - Every user sees the same AI selections (credibility/consistency)
    - Only 1 Grok API call per track-day regardless of user count (cost)
    - Near-instant response for 2nd+ user (performance)
    - Reduced FormFav scraping load (risk mitigation)

COMP / ADMIN ACCOUNTS
--------------------------------------------------------------------------------

  Server-side tier override — no StoreKit purchase needed.

  How it works:
    - Set tier to "pro" and subscriptionExpiresAt to null on the user record
    - Backend returns Pro status, app enforces no limits
    - No App Store transaction involved, purely a database flag

  Use cases:
    - Your own account (admin)
    - Beta testers during development
    - Influencer / promo giveaways
    - Competition prizes

  Admin endpoint (protected):

    POST /api/admin/set-tier
      - Header: Authorization: Bearer <ADMIN_SECRET> (env var on Vercel)
      - Body: { appleUserId, tier: "pro"|"basic"|"expired", expiresAt: null|ISO }
      - Updates user record directly
      - Optional expiresAt for time-limited comps (e.g. 30-day free Pro)

  Notes:
    - Keep ADMIN_SECRET in Vercel env vars, never in client code
    - Could build a simple admin web page later, but curl/Postman works for now
    - These accounts show as Pro in-app — no visual difference to the user


VERCEL BACKEND CHANGES
--------------------------------------------------------------------------------

  New endpoints:

    POST /api/auth/apple
      - Receives Apple identity token + user identifier
      - Validates token with Apple (verify signature + claims)
      - Creates or retrieves user record
      - Returns: { userId, tier, trialStartDate, usage }

    GET /api/user/status
      - Header: Authorization: Bearer <userId>
      - Returns current tier, usage counts, limits
      - App calls this on launch + before analysis

    POST /api/user/record-usage
      - Called after successful analysis
      - Body: { trackSlug, date }
      - Server records usage, returns updated counts

    POST /api/apple-notifications
      - App Store Server Notifications V2 webhook
      - Updates user tier based on subscription events
      - Handles: subscribe, renew, expire, revoke, refund

  New infrastructure:
    - Vercel KV or Upstash Redis for race analysis cache
    - User storage: Vercel Postgres (free tier) or Upstash Redis
      (Redis is simpler for key-value user records, Postgres is
      better if you need queries/analytics later)

EDGE AI UX FLOW (SIMPLIFIED FOR PAID MODEL)
--------------------------------------------------------------------------------

  Current behaviour (pre-subscription):
    - User taps Edge AI -> sheet with two options:
        "Analyse New Tracks" (only tracks without data today)
        "Re-analyse All Tracks" (refresh all selected tracks)
        Cancel
    - Track selection managed in TrackSelectorView (Settings)

  New behaviour (subscription model):
    - Remove the two-option sheet
    - User taps Edge AI -> single confirmation alert:
        "Run Edge AI?"
        "Run Edge AI Analysis on your selected tracks to generate
         picks and insights."
        [Cancel] [Run Analysis]
    - On confirm: analyses all tracks selected in TrackSelectorView (Settings)
    - Backend handles cache logic transparently — user doesn't need to know

  Cache-aware scrape logic (server-side, invisible to user):
    - App sends the user's selected tracks to backend
    - Backend checks cache for each track:
        CACHED  -> Returns stored result (no scrape, no Grok call)
        NEW     -> Scrapes FormFav, runs Grok, stores in cache, returns result
    - Response includes mix of cached + freshly scraped data
    - Usage only counted for tracks not already in the user's usage log
      (re-running same day = free, no double-counting)

  Basic tier (10 track-days/week):
    - If selected tracks would exceed the weekly limit:
        Show alert: "You have 3/10 analyses remaining this week.
         You've selected 8 meetings. Upgrade to Pro for unlimited access."
        [Upgrade to Pro] [Analyse 3 Tracks]
    - "Analyse 3 Tracks" runs analysis for only the first 3 uncounted tracks
      (tracks already used today don't count against the limit)

  Trial tier:
    - Same logic but with 3 total track-day limit
    - If would exceed: show paywall instead

  TrackSelectorView.swift:
    - Still used in Settings for track selection (user picks which tracks to follow)
    - No longer part of the Edge AI prompt — selection is pre-configured

CODE REVIEW NOTES — MIGRATION CONSIDERATIONS
--------------------------------------------------------------------------------

  1. forceFullScrape parameter removal
     - DataService.refreshScrape() currently takes forceFullScrape: Bool
       which controls whether to skip locally cached tracks
     - With server-side caching, the backend decides what's cached vs fresh
     - The simplified Edge AI UX (single "Run Analysis" button) removes the
       two-option modal, so this param is no longer needed
     - Remove forceFullScrape param; always send selected tracks, let backend
       handle cache hits vs misses

  2. Dual cache layers (both still needed)
     - Server-side cache (Vercel KV): deduplicates scrapes across ALL users,
       ensures consistent AI picks. Keyed by "track_date".
     - Local cache (device filesystem): stores race data for offline browsing
       and historical date navigation. mergeCacheData() logic stays as-is.
     - No conflict — server cache is cross-user, local cache is per-device.

  3. AnalysisService runs on-device — NOT gated
     - processRaceData() runs AnalysisService.shared.analyze() locally to
       compute BetSuggestions from raw race data
     - This is separate from the Grok AI analysis (server-side)
     - Decision: leave ungated. Local analysis adds no server cost and runs
       on cached data. Only the server-side Grok AI scrape is subscription-gated.

  4. TAB API calls — NOT gated
     - fetchTABOdds(), fetchRaceResults(), fetchTABSchedule() hit TAB
       directly from the user's device (AU geo-bypass)
     - These are free, no server cost, and support browsing/odds/results
     - Keep ungated for all tiers including expired users

  5. My Bets + Performance — NOT gated
     - MyBetsView and PerformanceView use SwiftData @Query on BetRecord
     - Fully local, no server involvement
     - Always accessible regardless of subscription tier
     - Users can view bet history, P&L, and performance stats even when expired

  6. Server Logs section — hide for public release
     - SettingsView currently shows "Server Logs" with live scrape output
     - For App Store release, hide this from regular users
     - Options: remove entirely, or gate behind a hidden debug gesture
       (e.g. tap version label 7 times)

  7. TrackSelection uses UserDefaults — stays local
     - TrackSelection.selectedSlugs stored in UserDefaults (device-local)
     - For subscription model, selected tracks are sent with each API request
       so the backend can validate usage counts
     - No need to sync track selection to the server

  8. 1-hour-before-first-race guard (already implemented)
     - DataService.refreshScrape() checks TAB schedule for each track's
       earliest race start time
     - Tracks whose first race is more than 1 hour away are excluded from
       the scrape and not sent to the server
     - A popup alerts the user: "[Track Names] not analysed. AI Analysis is
       available 1 hour before the first race to ensure up-to-date data
       is utilised."
     - If some tracks are eligible and others are too early, the eligible
       tracks are still analysed and the popup shows for the skipped ones
     - TABScheduleResult includes firstRaceTime: [String: Date] for this check
     - DataService publishes tooEarlyTrackNames: [String] consumed by
       TodayRacesView to show the alert

  9. File rename: GrokSelectionsView.swift -> AISelectionsView.swift
     - File is still named GrokSelectionsView.swift but the struct inside
       is AISelectionsView
     - Rename file to match struct before release for consistency
     - Also update any Xcode project references
